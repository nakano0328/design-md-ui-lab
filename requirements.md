# requirements.md

## 目的

このプロジェクトは、DESIGN.md をAIコーディングに読ませたときに、UIの品質や一貫性がどれくらい変わるかを試すためのフロントエンド実験サイトである。

実装対象は「AI導入提案ダッシュボード」の静的モックサイトとする。
実際の認証、DB、API連携、AI処理は実装しない。
UIがブラウザで確認できればよい。

この requirements.md は、Phase 1〜7 に分けているが、これは実装順序を明確にするためである。
ユーザーから「すべて実装して」と指示された場合は、Phase 1 から Phase 7 までを順番に一括で実装する。
ユーザーから「Phase XXだけ」と指示された場合のみ、そのPhaseだけを実装する。

---

## 作るサイトの概要

サイト名は `Proposal AI Dashboard` とする。

AI導入を検討している企業向けに、提案内容、導入ステップ、想定効果、案件ステータスを見える化するSaaS風ダッシュボードを作る。

主な画面は以下とする。

1. トップページ / LP
2. ダッシュボード画面
3. 提案詳細画面
4. 案件一覧画面
5. 設定画面

---

## 技術スタック

原則として以下を使う。

* Vite
* React
* TypeScript
* Tailwind CSS
* shadcn/ui は使用してもよい
* lucide-react は使用してもよい

バックエンドは作らない。
データは `src/data/mock.ts` のようなモックデータで管理する。

---

## デザイン方針

プロジェクトルートに `DESIGN.md` が存在する場合、必ずそれを参照してUIを実装する。

### 重要ルール

* `DESIGN.md` の色、余白、タイポグラフィ、ボタン、カード、フォーム、ナビゲーション方針を優先する
* ただし、可読性とレスポンシブ対応を壊してはいけない
* UIの完成度を確認しやすいように、画面内にカード、テーブル、フォーム、バッジ、グラフ風パーツを含める
* 実装は静的モックでよい
* 画像生成は不要
* 外部API連携は不要

# Phase 1: プロジェクト初期化と基本レイアウト

## 目的

React + TypeScript + Tailwind CSS の基本環境を作り、ブラウザで表示できる状態にする。

## 実装内容

* Vite + React + TypeScript のセットアップ
* Tailwind CSS のセットアップ
* `src/App.tsx` を作成
* `src/styles.css` を作成
* `LandingPage` を仮表示
* Header を作成
* 全体の背景、文字、余白を整える
* `DESIGN.md` がある場合は、色・余白・角丸・フォント感を反映する

## 完了条件

* `npm install`
* `npm run dev`

でローカル表示できる。

トップページに以下が表示される。

* サービス名
* キャッチコピー
* CTAボタン
* 簡単な説明カード3つ

---

# Phase 2: LPを完成させる

## 目的

DESIGN.md の効果が分かりやすいLPを作る。

## 実装内容

トップページに以下のセクションを作る。

1. Hero

   * サービス名
   * キャッチコピー
   * 説明文
   * CTAボタン2つ
2. Problem

   * AI導入前の課題カード3つ
3. Solution

   * 提案作成、効果試算、案件管理の説明カード3つ
4. Workflow

   * 相談 → ヒアリング → 提案作成 → 導入判断 の4ステップ
5. Demo Preview

   * ダッシュボードのプレビューカード
6. Footer

## UI要件

* カード、ボタン、バッジ、アイコンを使う
* レスポンシブ対応する
* PC表示で見栄えがすることを優先する
* スクロールして見たときに、SaaS LPらしく見えること

## 完了条件

LPとして一通り見える状態になる。

---

# Phase 3: ダッシュボード画面を作る

## 目的

管理画面UIの見た目を確認できるようにする。

## 実装内容

`DashboardPage` を作る。

画面には以下を含める。

* Sidebar
* Header
* KPIカード4つ

  * 提案中案件数
  * 今月の想定売上
  * 平均ROI
  * 次回アクション数
* 案件ステータス一覧
* 効果試算のグラフ風カード
* 最近のアクティビティ
* 優先対応リスト

## UI要件

* 実際のグラフライブラリは使わなくてよい
* CSSだけで棒グラフ風、折れ線グラフ風、進捗バー風の表現を作る
* ステータスは色付きバッジで表現する
* データはモックでよい

## 完了条件

SaaSの管理画面らしいUIになっている。

---

# Phase 4: 提案詳細画面を作る

## 目的

1つの提案内容を詳細に確認できる画面を作る。

## 実装内容

`ProposalDetailPage` を作る。

画面には以下を含める。

* 提案タイトル
* 企業名
* ステータス
* 提案概要
* 導入ステップ
* 想定効果
* リスク・確認事項
* 次回アクション
* 担当者コメント欄風UI

## UI要件

* タブUIまたはセクション分割を使う
* 重要情報が見やすいようにカード化する
* コメント欄は静的表示でよい
* 編集機能は不要

## 完了条件

営業提案やAI導入支援の管理画面として自然に見える。

---

# Phase 5: 案件一覧画面を作る

## 目的

複数案件を一覧で確認できるテーブルUIを作る。

## 実装内容

`ProjectsPage` を作る。

画面には以下を含める。

* 検索欄
* フィルタボタン
* 案件一覧テーブル
* ステータスバッジ
* 優先度表示
* 担当者表示
* 最終更新日
* 詳細を見るボタン

## UI要件

* 実際の検索・フィルタ処理は簡易でよい
* 見た目として操作できそうに見えればよい
* テーブルがPCで見やすいこと
* スマホではカード表示に切り替えてもよい

## 完了条件

SaaSの案件管理画面として使えそうに見える。

---

# Phase 6: 設定画面と最終調整

## 目的

フォームUI、トグル、選択UIなどを確認できる画面を作り、全体の完成度を上げる。

## 実装内容

`SettingsPage` を作る。

画面には以下を含める。

* プロフィール設定
* 通知設定
* 提案テンプレート設定
* デザインモード設定風UI
* 保存ボタン

加えて以下を行う。

* ページ遷移またはタブ切り替えを実装
* 全体の余白・色・カードの統一
* レスポンシブ調整
* README作成

## 完了条件

LP、ダッシュボード、提案詳細、案件一覧、設定画面を移動して確認できる。

# Phase 7: DESIGN.md比較用の仕上げ

## 目的

## 実装内容

* 主要コンポーネントを1ページで確認できる `ComponentShowcasePage` を作る
* 以下のUI部品を一覧表示する

  * ボタン
  * カード
  * バッジ
  * フォーム
  * テーブル
  * KPIカード
  * アラート
  * ナビゲーション
* READMEに、DESIGN.mdを差し替えて再実装・再調整する手順を書く

## 完了条件

DESIGN.mdを変えたときの差分が見やすいデモサイトになる。

---

## 実装時の注意

* バックエンドを作らない
* 認証を作らない
* DBを作らない
* API通信を作らない
* UIの完成度を最優先する
* 各Phaseでは、そのPhaseに必要な範囲だけ変更する
* 既存の実装を壊さない
* 迷ったらDESIGN.mdの方針を優先する
